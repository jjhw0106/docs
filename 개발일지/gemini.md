# 🤖 Gemini 작업 가이드

이 문서는 Gemini가 프로젝트 내에서 개발일지를 작성하고 할 일을 관리하는 표준 원칙을 정의합니다.

---

## 📝 1. 개발일지 작성 (Journaling)

작업한 모든 내용은 기록으로 남겨 히스토리를 추적할 수 있도록 합니다.

### 📍 작성 위치 및 구조
- **기본 경로**: `my-docs/개발일지/`
- **프로젝트별 분류**:
  - `front/`: 프론트엔드 관련 작업 기록
  - `api/`: 백엔드/API 관련 작업 기록
  - 그 외 적절한 프로젝트 디렉토리 하위에 작성

### 🏷️ 네이밍 규칙
- **형식**: `YYMMDD_프로젝트명.md`
- **예시**: `251227_프론트.md`, `260217_로또_백엔드.md`

### ✍️ 작성 가이드
- 기존 파일의 양식을 참고하여 일관된 스타일 유지.
- 작업의 목표, 구현 내용, 트러블슈팅, 회고 등을 포함.

---

## 🎯 2. To-Do 리스트 관리 (Task Management)

프로젝트의 진행 상황을 명확히 하고 다음 단계를 계획합니다.

### 📍 관리 파일
- `my-docs/To-Do.md`

### ✍️ 작성 원칙
- **문체**: 존댓말이 아닌 **명사형 종결** 문장 사용 (예: ~함, ~진행).
- **날짜 기록**: 파일 최상단에 **최종 작업 날짜**를 기입 (날짜는 추가하지 않고 항상 최신 기준으로 업데이트).
- **구조화**: BE(Back-End)와 FE(Front-End)에서 할 일을 구분하여 작성.
- **내용 구성**:
  1. 사용자가 요청한 사항을 우선적으로 반영.
  2. Gemini가 판단했을 때 필요한 기술적 부채나 개선 사항 추가.
- **진행 상황 표시**:
  - 해결한 항목: `- ~~내용~~` (취소선 처리)
  - 완료된 항목: `- [x] ~~내용~~` (체크박스 마킹)

---

## 💾 3. 버전 관리 및 커밋 (Git Commit)

안전하고 명확한 커밋을 위한 절차를 준수합니다.

### 📜 커밋 메시지 작성
- **멀티라인 처리**: 메시지가 길 경우 반드시 별도의 텍스트 파일(예: `commit_msg.txt`)에 작성한 뒤 `git commit -F` 옵션 사용.
- **임시 파일 정리**: 커밋 완료 후 생성한 임시 텍스트 파일은 반드시 즉시 삭제 (`rm`).

### 🚀 푸시 및 정렬
- 작업 완료 후 지체 없이 푸시하여 원격 저장소와 동기화.
- 전체 프로젝트 내에 불필요한 임시 파일이 남지 않도록 항상 체크.
