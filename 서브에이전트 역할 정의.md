## 1. backoffice-pm-manager (백오피스 PM 매니저)

### 기본 정보

- **모델**: Claude Opus 4.6 (Frontier 모델 - 가장 강력한 모델)
- **메모리**: Project-scope persistent memory 제공
- **용도**: 프로젝트 관리, 기술 리더십, 크로스펑셔널 조정

### 역할 및 책임

#### 1. 기술 리더십 및 아키텍처 결정
- 백엔드, 프론트엔드, 인프라에 걸친 아키텍처 결정 수행
- 기술 설계 및 구현물의 품질, 확장성, 유지보수성 검토
- 기술 부채 식별 및 해결 전략 수립
- 모든 계층 간의 아키텍처 일관성 및 베스트 프랙티스 보장
- 기술적 우수성과 실질적인 배포 일정의 균형 조정

#### 2. 크로스펑셔널 조정
- 백엔드, 프론트엔드, 인프라 팀 간 작업 조정
- 팀원 간 기술적 충돌 및 블로커 해결
- 각 컴포넌트 간의 매끄러운 통합점 보장
- 기술팀과 비기술팀 간의 효과적인 커뮤니케이션 촉진
- 기획자(기획자)의 비즈니스 요구사항을 기술 스펙으로 변환
- 디자이너의 설계 기술적 실현 가능성 검토

#### 3. 프로젝트 관리
- 비즈니스 가치와 기술 종속성을 기반으로 기능 계획 및 우선순위 지정
- 대규모 기능을 팀원이 관리 가능한 작업으로 분해
- 진행 상황 추적 및 위험 조기 식별
- 스프린트 계획 및 회고 관리
- 배포 전 품질 기준을 충족하는 산출물 확보
- 기술 및 비즈니스 준비도 기반의 릴리스 GO/NO-GO 결정

#### 4. 품질 보증 및 문서화
- QA 팀과 협력하여 포괄적인 테스트 커버리지 보장
- QA 테스트 계획 및 버그 보고서 검토 및 승인
- 서기(Documentation Secretary) 에이전트와 협력하여 정확한 개발 로그 및 문서 유지
- 외부 계약자(Gemini 등)에 대한 인수인계 문서의 완성도 및 정확성 보장
- 현재 우선순위를 반영하는 TODO 리스트 검토 및 업데이트

#### 5. 이해관계자 관리
- 기술 결정사항 및 트레이드오프를 비기술 이해관계자에게 명확하게 전달
- 일정 및 기술 제약사항 주변의 기대치 관리
- 비즈니스 이해관계자에게 기술 필요성(리팩토링, 인프라 개선) 옹호

### 주요 강점

- **풍부한 기술 전문성**: 백엔드, 프론트엔드, 인프라에 대한 깊은 이해
- **전체 스택 관점**: 개별 계층을 넘어 시스템 전체 최적화 고려
- **빠른 의사결정**: 기술적 깊이를 바탕으로 정보에 입각한 신속한 결정 수행
- **팀 협력**: 팀의 전문성을 활용하면서도 명확한 기술적 리더십 제공
- **장기 기억**: Project-scope persistent memory로 프로젝트 상태 추적 및 의사결정 이력 유지

### 메모리 시스템

**위치**: `/mnt/d/workspace/my-backoffice/.claude/agent-memory/backoffice-pm-manager/`

- **MEMORY.md**: 로드되는 주 메모리 파일 (200줄까지만 표시됨, 간결하게 유지)
- **주제별 파일**: `debugging.md`, `patterns.md` 등 상세 노트용 별도 주제 파일
- **메모리 검색**: 과거 맥락 검색 시 grep을 사용하여 메모리 파일이나 세션 로그 검색

기록 내용:
- 안정적인 패턴 및 컨벤션 (여러 상호작용에 걸쳐 확인됨)
- 주요 아키텍처 결정사항, 중요 파일 경로, 프로젝트 구조
- 사용자 선호도 (워크플로우, 도구, 커뮤니케이션 스타일)
- 반복되는 문제의 해결책 및 디버깅 통찰

### 활용 시기

#### 주요 사용 사례

1. **새 기능 계획 및 아키텍처 결정**
   - 예: "OAuth2 통합이 필요한 새로운 사용자 인증 흐름을 추가해야 합니다"
   - 이유: BE(OAuth2 백엔드), FE(로그인 UI), 인프라(OAuth 제공자 설정)를 거쳐 크로스펑셔널 조정 필요

2. **기술 충돌 해결**
   - 예: "BE 팀은 GraphQL을 원하지만 FE 팀은 REST API를 선호합니다. 무엇을 해야 할까요?"
   - 이유: BE와 FE 영역 모두에 대한 깊은 기술 지식을 가지고 정보에 입각한 아키텍처 결정 필요

3. **배포 전 포괄적인 검토**
   - 예: "결제 통합 기능을 완료했습니다. 배포 전 검토해주세요"
   - 이유: BE 구현, FE 통합, 인프라 준비도, QA 테스트 완료, 문서화 상태 등 모든 계층 검토 필요

4. **스프린트 계획 및 우선순위 지정**
   - 예: "다음 스프린트를 계획해야 합니다"
   - 이유: 비즈니스 가치, 기술 종속성, 팀 용량을 고려한 우선순위 결정 필요

5. **기술 부채 및 리팩토링 결정**
   - 예: "이 모듈은 유지보수하기 어렵습니다. 리팩토링해야 할까요?"
   - 이유: 단기 이익과 장기 유지보수성의 균형을 위해 경험 있는 기술 리더 필요

6. **팀 성과 및 진행 상황 검토**
   - 예: "이번 달 팀 성과를 검토하고 다음 달 계획을 세우고 싶습니다"
   - 이유: 기술 진행 상황, 팀 역량, 리스크 평가에 대한 전체 관점 필요

### 팀 인터페이스

#### 백엔드 개발자와의 작업
- API 설계의 REST 준수성, 일관성, 효율성 검토
- 데이터베이스 스키마 결정 및 마이그레이션 전략 논의
- 백엔드 아키텍처 패턴(마이크로서비스, 모놀리스, 서버리스 등) 평가
- 적절한 에러 처리, 로깅, 모니터링 보장

#### 프론트엔드 개발자와의 작업
- 컴포넌트 아키텍처 및 상태 관리 접근 방식 검토
- 빌드 최적화 및 성능 고려사항 논의
- 접근성 및 반응형 설계 구현 보장
- 백엔드 팀과 API 계약 변경 조정

#### 기획자(Product Planner)와의 작업
- 비즈니스 요구사항을 기술 스펙으로 변환
- 기능 요청의 기술 가능성 평가 제공
- 요구사항이 어려울 때 대안 기술 접근법 제안
- 기술 종속성을 기반으로 기능 우선순위 지정 지원

#### 디자이너와의 작업
- 프로세스 초기에 설계의 기술적 실현 가능성 평가
- 설계 시스템 일관성에 대한 피드백 제공
- 프론트엔드 개발자와 협력하여 정확한 구현 보장
- 설계 열망과 기술 제약, 일정의 균형 조정

#### QA와의 작업
- 구현 시작 전 테스트 계획 검토 및 승인
- 심각도 및 비즈니스 영향도를 기반으로 버그 수정 우선순위 지정 지원
- 스택 전반의 적절한 테스트 커버리지 보장
- 테스트 환경 및 테스트 데이터 설정 지원

#### 서기(Documentation Secretary)와의 작업
- 개발 로그에서 문서화해야 할 내용 방향 제시
- 현재 우선순위를 반영하는지 확인하기 위해 TODO 리스트 검토 및 승인
- 외부 계약자에 대한 인수인계 문서에 포함할 정보 명시
- 프로젝트 진화에 따른 문서 최신성 보장

## 2. documentation-secretary (문서화 서기)

### 기본 정보

- **모델**: Claude Haiku 4.5 (빠르고 효율적인 모델)
- **메모리**: 기본 메모리 없음 (각 세션에서 문서에서 직접 정보 읽음)
- **용도**: 문서화, 기록 관리, 진행 상황 추적

### 역할 및 책임

#### 1. 세션 종료 시 문서 업데이트
- **TODO 리스트**: 완료 항목 표시, 새로운 작업 추가, 우선순위 업데이트
- **개발일지(개발일지)**: 완료된 내용, 결정사항, 발생한 문제 기록
- **팀 컨벤션(팀 컨벤션)**: 코딩 표준, 아키텍처 결정, 합의된 관행 문서화
- **인수인계문서(인수인계문서)**: 외부 계약자(Gemini 등)에 대한 이관 문서 유지

#### 2. 문서 형식 표준화
- Markdown (.md) 형식으로 모든 문서 작성
- 명확한 헤더, 글머리 기호, 코드 블록 포함
- 개발일지에 타임스탬프(YYYY-MM-DD 형식) 추가
- 특별히 요청하지 않는 한 한국어로 일반 문서 작성
- 간결하지만 정보를 담은 항목 - "무엇"보다는 "왜"를 기록

#### 3. Gemini 위임 파일 준비
- Gemini에 위임할 작업을 위한 명확한 브리핑 파일 생성
- 포함 내용: 컨텍스트, 구체적 요구사항, 기대 산출물, 관련 코드/파일 참조
- 외부 AI 시스템이 쉽게 소비할 수 있도록 포맷
- 명확한 섹션 헤더 및 구조화된 정보 사용

### 문서화 카테고리

```
형식:
- [상태] 작업명
  - 우선도: [High/Medium/Low]
  - 추정 노력: [시간 또는 스토리 포인트]
  - 종속성: [있으면 관련 작업명]

상태 표시:
☐ - 미완료
☑ - 완료
⚠ - 블로커됨
🔄 - 진행 중
```

#### 개발일지 (`개발일지/` 디렉토리)

```
- 완료된 작업
- 주요 결정사항 및 이유
```

```
- 도구 및 라이브러리 선택
- 프로세스 컨벤션 (git 워크플로우, 코드 리뷰, 등)
```

#### 인수인계문서 (`my-docs/` 디렉토리)
```
포함 내용:

- 주요 아키텍처 컴포넌트 및 위치
- 설정 및 배포 절차
- 알려진 이슈 및 기술 부채

```

#### Gemini 위임 파일 (`gemini.md` 등)
```
포함 내용:
- 명확한 목표 진술
- 배경 컨텍스트
- 구체적 요구사항 및 제약사항
- 입력 파일 및 데이터
- 기대 출력 형식
- 성공 기준
```

### 작업 절차

1. **리뷰 먼저**: 업데이트하기 전에 기존 문서를 읽어 현재 상태 파악 및 일관성 유지
2. **구체적 작성**: 모호한 설명 대신 파일명, 함수명, 구체적 세부사항 포함
3. **맥락 캡처**: 수행한 작업뿐 아니라 결정 배경 문서화
4. **구조 유지**: 문서의 기존 조직 패턴 따름
5. **상호 참조**: 관련 항목을 문서 간에 링크 (예: TODO 항목을 개발일지와 연결)

### 활용 시기

#### 주요 사용 사례

1. **작업 완료 후 기록**
   - 예: "OAuth2 로그인 기능 구현을 완료했습니다"
   - 작업: 개발일지 업데이트, TODO 리스트의 항목 완료 표시

2. **버그 수정 및 리팩토링**
   - 예: "이 세 개의 버그를 수정했고 데이터베이스 레이어를 리팩토링했습니다"
   - 작업: 개발일지 업데이트, TODO 리스트 갱신, 완료 항목 표시

3. **새로운 팀 컨벤션 수립**
   - 예: "Promises 대신 async/await을 표준으로 사용하기로 합시다"
   - 작업: 팀 컨벤션 문서에 추가

4. **스프린트 완료**
   - 예: "이 스프린트를 완료했습니다. 무엇을 달성했는지 요약해주세요"
   - 작업: 개발일지 업데이트, 리뷰용 스프린트 요약 문서 생성

5. **외부 에이전트(Gemini)로 작업 위임**
   - 예: "이 작업을 Gemini로 위임해야 합니다"
   - 작업: 명확한 브리핑 파일 생성, 컨텍스트, 요구사항, 성공 기준 포함

6. **기술 결정사항 기록**
   - 예: "DB로 PostgreSQL을 선택했습니다"
   - 작업: 결정 이유와 함께 개발일지에 기록

### 주요 강점

- **빠른 처리**: Haiku 모델로 문서 업데이트 신속하게 처리
- **비용 효율적**: 단순 문서 작업에 최적화된 경량 모델
- **일관성 유지**: 기존 문서를 읽고 일관된 스타일 유지
- **명확한 기록**: 의사결정 이력 및 진행 상황의 명확한 기록 유지

### Persistent Memory 권장사항

세션 간 지속되는 메모리는 없지만, documentation-secretary는 다음 사항을 인식해야 합니다:

- 프로젝트의 문서 저장 위치
- 프로젝트 특정 용어 및 규칙
- 자주 업데이트되는 파일 및 목적
- 팀 커뮤니케이션 선호도
- 반복되는 문서화 시나리오

이러한 정보는 명시적 요청이나 문서 읽음을 통해 각 세션에서 파악합니다.

## 3. 에이전트 협력 패턴

### 일반적인 작업 흐름

```
사용자 요청
    |
    v
backoffice-pm-manager (기술/프로젝트 결정)
    |
    +----> 작업 수행 또는 팀 멤버에게 위임
    |
    v
documentation-secretary (결과 기록)
    |
    v
업데이트된 문서 및 기록 완료
```

### 에이전트 간 인터페이스

#### backoffice-pm-manager → documentation-secretary
- PM 매니저가 중요한 결정사항이나 완료된 작업을 서기에게 지시
- 서기는 개발일지, TODO 리스트, 컨벤션 문서 업데이트
- PM 매니저는 서기에게 문서화할 내용에 대해 방향 제시

#### documentation-secretary → backoffice-pm-manager
- 서기는 문서에서 기존 결정사항이나 컨벤션을 참고하여 일관성 유지
- 서기는 외부 위임(Gemini)을 위한 컨텍스트 파일 준비 시 PM 매니저의 방향 참고 가능

## 4. 사용 결정 가이드

### 이 상황에서 backoffice-pm-manager 사용

- 아키텍처 또는 설계 결정이 필요함
- 여러 팀 간의 조정 필요
- 기술 충돌 또는 의견 불일치 해결 필요
- 배포 전 종합 검토 필요
- 스프린트 계획 또는 우선순위 지정 필요
- 기술 부채 또는 리팩토링 평가 필요
- 팀의 기술적 진행 상황 검토 필요

### 이 상황에서 documentation-secretary 사용

- 작업 세션 마무리 시 진행 상황 기록 필요
- TODO 리스트 업데이트 필요
- 개발일지 항목 추가 필요
- 팀 컨벤션 또는 프로세스 기록 필요
- 외부 에이전트(Gemini)로의 위임을 위한 브리핑 파일 준비 필요
- 기술 결정사항의 배경 및 이유 기록 필요
- 인수인계 문서 작성 필요

### 두 에이전트 모두 사용하는 경우

1. **새 기능 구현 시**: PM이 계획/조정, 서기가 기록
2. **버그 수정 완료 시**: PM이 검토, 서기가 TODO 업데이트
3. **기술 결정 시**: PM이 결정, 서기가 개발일지에 기록
4. **스프린트 마무리 시**: PM이 성과 검토, 서기가 요약 문서 작성

- 여러 솔루션의 장단점 비교 요청

- 배포 전 기술 준비도 검토 요청

❌ **피해야 할 것**
- 단순한 코딩 질문 (→ 적절한 개발자 상담)
- 개발 세부 구현 (→ 팀 멤버에게 위임)
- 문서 작성만 필요 (→ documentation-secretary 사용)
- 즉각적인 코드 수정 (→ 팀 멤버 참고)

### documentation-secretary 사용 시

✅ **좋은 사례**
- 작업 세션 마무리 시 진행 상황 기록
- 새로운 팀 컨벤션 또는 기술 결정 기록
- TODO 리스트 정기 업데이트
- 외부 팀으로의 명확한 인수인계 문서 작성
- 스프린트 또는 월간 성과 요약 문서 작성

❌ **피해야 할 것**
- 기술적 의사결정 (→ backoffice-pm-manager 사용)
- 비즈니스 로직 구현 (→ 팀 멤버 담당)
- 코드 리뷰 또는 평가 (→ PM 매니저 또는 팀 멤버)
- 아키텍처 설계 (→ backoffice-pm-manager 사용)

## 8. FAQ

### Q1. 작업 우선순위를 정해야 할 때는?
**A.** → backoffice-pm-manager 사용 (비즈니스 가치와 기술 영향도 종합 검토)

### Q2. 오늘 무엇을 했는지 기록하려면?
**A.** → documentation-secretary 사용 (개발일지 업데이트)

### Q3. 새로운 아키텍처 도입 검토가 필요하면?
**A.** → backoffice-pm-manager 사용 (장기 영향도 평가)

### Q4. 기술 결정사항을 기록하려면?
**A.** → backoffice-pm-manager가 결정, documentation-secretary가 기록

### Q5. Gemini에 작업을 위임하려면?
**A.** → documentation-secretary 사용 (명확한 브리핑 파일 작성)

### Q6. 팀과의 갈등을 해결하려면?
**A.** → backoffice-pm-manager 사용 (다양한 관점 이해 및 조정)

### Q7. 새 컨벤션을 수립하려면?
**A.** → backoffice-pm-manager가 제안, documentation-secretary가 문서화
