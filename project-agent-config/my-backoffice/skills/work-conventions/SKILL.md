# Work Conventions Skill

my-backoffice 프로젝트의 작업 원칙, 커밋 가이드, 개발일지 규칙을 정의하는 스킬.

## 작업 원칙 (Working Principles)

- **선 설명 후 조치 (Explain First, Act Later)**: 코드를 수정하거나 파일을 생성하기 전에, **반드시** 구현 계획과 변경 내용을 사용자에게 먼저 설명한다.
- **사용자 승인 필수**: 사용자의 명확한 **동의(컨펌)**가 떨어진 후에만 실제 코드 변경을 진행한다.

## 커밋 실행 가이드 (Commit Execution Guide)

- **멀티라인 메시지 처리**: 커밋 메시지가 길거나 여러 줄(Multi-line)로 구성된 경우, 쉘 명령어 파싱 오류를 방지하기 위해 **반드시** 별도의 텍스트 파일(예: `commit_msg.txt`)에 메시지를 작성한 뒤 `git commit -F <파일명>` 옵션을 사용하여 커밋한다.
- **권장 절차**:
  1. `commit_msg.txt` 생성 (메시지 내용 작성)
  2. `git commit -F commit_msg.txt`
  3. `rm commit_msg.txt` (임시 파일 즉시 삭제)
- 커밋 완료 후 불필요한 임시 파일이 남지 않도록 항상 체크.

## 개발일지 작성 규칙 (Dev Journal)

- **기본 경로**: `docs/개발일지/`
- **프로젝트별 분류**:
  - `front/`: 프론트엔드 관련 작업 기록
  - `api/`: 백엔드/API 관련 작업 기록
  - 그 외 적절한 프로젝트 디렉토리 하위에 작성
- **네이밍 규칙**: `YYMMDD_프로젝트명.md` (예: `251227_프론트.md`, `260217_로또_백엔드.md`)
- **작성 가이드**: 기존 파일의 양식을 참고하여 일관된 스타일 유지. 작업의 목표, 구현 내용, 트러블슈팅, 회고 등을 포함.
