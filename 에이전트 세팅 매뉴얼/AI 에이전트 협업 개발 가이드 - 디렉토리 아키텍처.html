AI 에이전트 협업 개발 가이드 - 디렉토리 아키텍처

Claude Code + Gemini를 함께 쓰는 멀티 AI 개발 환경에서,
컨텍스트를 효율적으로 관리하고 토큰을 절약하기 위한 디렉토리 설계 원칙과 구조.

1. 왜 이 구조가 필요한가

AI 코딩 어시스턴트를 쓸 때 가장 큰 병목은 컨텍스트 오염이다.

문제증상
관련 없는 파일까지 컨텍스트에 포함토큰 낭비, 응답 품질 저하
설정이 각 서비스 repo 안에 분산AI 세팅 변경 시 여러 repo 수동 수정
Claude / Gemini 간 역할 경계 불명확작업 중복, 핸드오프 혼선
서비스별 컨텍스트를 항상 전부 로드불필요한 토큰 소모
해결 방향 — 설정을 docs 단일 레포에서 중앙 관리하고, 심링크로 각 프로젝트에 연결한다. AI는 필요한 컨텍스트만 골라 로드할 수 있게 된다.

4. 심링크 아키텍처

docs에서 설정을 한 번만 수정하면 모든 프로젝트에 즉시 반영된다. 각 서비스 repo 안에 .claude/를 두지 않는다.

📁 docs/project-agent-config/my-backoffice/  (원본 · git 추적)
CLAUDE.md agents/ skills/ prds/ handoffs/
SymbolicLink (심링크)
📁 project-my-backoffice/  (작업 디렉토리)
CLAUDE.md .claude/agents/ .claude/skills/ prds/ handoffs/

심링크 맵

프로젝트 내 경로타입소스 (docs)
project-{name}/CLAUDE.md파일 심링크project-agent-config/{name}/CLAUDE.md
project-{name}/.claude/agents/디렉토리 심링크project-agent-config/{name}/agents/
project-{name}/.claude/skills/디렉토리 심링크project-agent-config/{name}/skills/
project-{name}/prds/디렉토리 심링크project-agent-config/{name}/prds/
project-{name}/handoffs/디렉토리 심링크project-agent-config/{name}/handoffs/

심링크 생성 명령 (Windows PowerShell)

New-Item -ItemType SymbolicLink -Path "project-{name}\CLAUDE.md"      -Target "docs\project-agent-config\{name}\CLAUDE.md"
New-Item -ItemType SymbolicLink -Path "project-{name}\.claude\agents" -Target "docs\project-agent-config\{name}\agents"
New-Item -ItemType SymbolicLink -Path "project-{name}\.claude\skills" -Target "docs\project-agent-config\{name}\skills"
New-Item -ItemType SymbolicLink -Path "project-{name}\prds"           -Target "docs\project-agent-config\{name}\prds"
New-Item -ItemType SymbolicLink -Path "project-{name}\handoffs"       -Target "docs\project-agent-config\{name}\handoffs"
사전 조건 — Windows 설정 → 개발자 모드 ON 또는 관리자 권한 터미널

5. 컨텍스트 로딩 전략

작업 복잡도에 따른 로딩 흐름

Claude 세션 시작
        │
        ▼
  CLAUDE.md 자동 로드 (프로젝트 전체 구조, 포트, 기술스택, 오케스트레이터 원칙)
        │
        ├─ 단순 작업?       → Main이 직접 처리
        ├─ 중간 복잡도?     → be-agent / fe-agent 로드 후 위임
        └─ 복잡/신규 기능?  → /handoff-generator → Gemini 핸드오프

서비스별 컨텍스트 분리 (토큰 절약 핵심)

작업: lotto-server 기능 추가
        ├─ 로드: project CLAUDE.md
        ├─ 로드: service-context/lotto-server/CLAUDE.md
        └─ 로드: lotto-domain/SKILL.md

        미로드: auth-server / gateway / scraper-server / frontend 컨텍스트

7. 스킬 시스템

8. 핸드오프 문서 흐름

1
핸드오프 생성 Main Claude → /handoff-generator → handoffs/{feature}/handoff-v1.md 생성
2
Gemini 구현 핸드오프 문서 기반으로 구현, 질문은 QUESTIONS 섹션에 추가
3
Q&A 사이클 QUESTIONS → ANSWERS(Main) → CONFIRMED(Gemini) → RESOLVED(Main만)
4
코드 리뷰 be-agent / fe-agent → /code-reviewer 스킬로 Gemini 코드 리뷰
5
최종 승인 Main Claude + 사용자 최종 컨펌
6
완료 처리 handoffs/ → reviewed/ 이동, 파일명에 -final 추가
